## 2. 기술 아키텍처

> 아키텍처 다이어그램 및 프로젝트 파일 구조는 [0.overview.md — 3절·6절](0.overview.md#3-아키텍처) 참조.

### 2.1 기술 스택

| 계층 | 기술 | 선택 이유 |
|------|------|---------|
| MCP 런타임 | @modelcontextprotocol/sdk | 공식 SDK, stdio transport |
| 스키마 검증 | Zod | 도구 파라미터 타입 안전성 |
| AI 분석 | Claude API (fetch) | BYOK, 서버 불필요 |
| 저장소 | 프로젝트별 JSON 파일 (fs) | 프로젝트 단위 I/O, 백업/공유 용이 |
| 테스트 | node:test (내장) | 외부 의존성 없음 |
| 런타임 | Node.js 18+ | ES Modules, 내장 fetch |

### 2.2 데이터 모델

**HistoryEntry (v0.4.0 — grade, missingElements, analysisMode 추가):**

```javascript
{
  id:              string,    // h_timestamp_random
  projectId:       string,
  prompt:          string,    // 원본 프롬프트
  enhanced:        string,    // 보강된 프롬프트
  score:           number,    // 0-100
  grade:           string,    // A/B/C/D (score 기반 캐시, 쿼리 최적화)
  axisScores:      number[],  // [clarity, specificity, context, structure, actionability]
  missingElements: string[],  // 누락 요소 목록 (query_history 필터용)
  analysisMode:    string,    // "local" | "api"
  tags:            string[],
  note:            string,
  platform:        string,    // "claude"
  parentId:        string|null,
  version:         number,
  createdAt:       string,    // ISO 8601
  updatedAt:       string
}
```

**Settings:**

```javascript
{
  apiKey:   string,   // Anthropic API 키 (로컬 저장)
  model:    string    // 기본: claude-sonnet-4-5-20250514
}
```

## 5. v0.5.1 개발 계획

### 5.1 목표

반복 마찰을 제거하여 프롬프트 개선 루프 속도를 높인다. v0.5.0이 분석 품질을 강화했다면, v0.5.1은 **사용자 편의성**에 집중한다. 상세 기능 명세는 `1.add.md` 4절 참조.

| 기능 | 유형 | 내용 |
|------|------|------|
| 세션 컨텍스트 (`>> re`) | 인프라 | 인메모리 lastEntryId 추적으로 ID 자동 참조 |
| 자동 태그 추천 | `analyze_prompt` 확장 | local(규칙) / api(Claude) 태그 제안 |
| 배치 분석 (`batch_analyze`) | 신규 도구 | 최대 20개 프롬프트 일괄 분석 |
| 자동 개선 생성 (`>> fix`) | 신규 내부 함수 + 트리거 | missingElements 기반 개선본 자동 생성 |
| 자동 개선 루프 (`>> loop`) | 신규 트리거 | 목표 점수까지 fix→analyze 자동 반복 |
| MCP Prompts 프리셋 확장 | 서버 설정 | `/` 메뉴에 도메인별 프리셋 3종 추가 (code-review, doc-writing, system-design) |
| 개선 제안 클립보드 포맷 | `analyze_prompt` 응답 개선 | `enhancedPromptBlock` 필드 추가 — 구분선으로 감싼 복사 가능 블록 |

### 5.2 구현 순서

기능 간 의존 관계에 따라 순서를 고정한다:

```
Phase 1 (독립 구현)
  ├── 세션 컨텍스트 (>> re / >> v2)   ← 인메모리, 의존성 없음
  └── 자동 태그 추천                  ← analyze_prompt 확장, 독립

Phase 2 (analyze_prompt 안정화 후)
  └── 배치 분석 (batch_analyze)       ← analyzePrompt() 내부 로직 재사용

Phase 3 (Phase 2 완료 후)
  └── 자동 개선 생성 (>> fix)         ← Claude API 호출 + parentId 연결

Phase 4 (Phase 3 완료 후)
  └── 자동 개선 루프 (>> loop)        ← fix 내부 로직 의존 필수
```

### 5.3 구현 일정 (4주)

| 주차 | Phase | 산출물 |
|------|-------|--------|
| 1주 | Phase 1 | `session.js` 모듈 (lastEntryId 관리), `>> re` / `>> v2` 트리거, `analyzer.js`에 `autoTag` / `tagMode` 추가, local 태그 규칙 테이블, **`enhancedPromptBlock` 응답 필드 추가 (analyzer.js)** |
| 2주 | Phase 2 | `batch_analyze` 도구 등록, 순차(local) / 병렬(api, semaphore 3) 처리, 점수 내림차순 정렬 반환, **MCP Prompts 프리셋 3종 등록 (index.js `server.prompt()`)** |
| 3주 | Phase 3 | `improver.js` 신규 모듈, `>> fix` / `>> fix+save` 트리거, parentId 자동 연결 저장 |
| 4주 | Phase 4 + 통합 | `>> loop N` 트리거, maxIterations 가드, E2E 테스트, 문서 업데이트, v0.5.1 릴리스 |

### 5.4 도구 변경

| 유형 | 도구 | 내용 |
|------|------|------|
| 신규 | `batch_analyze` | 최대 20개 일괄 분석, 결과 순위 테이블 반환 |
| 수정 | `analyze_prompt` | `autoTag`, `tagMode` 파라미터 추가, `suggestedTags` / `enhancedPromptBlock` 응답 필드 추가 |
| 트리거 추가 | — | `>> fix`, `>> fix+save`, `>> re`, `>> v2`, `>> loop`, `>> loop N` |
| 서버 설정 추가 | `server.prompt()` | `code-review`, `doc-writing`, `system-design` 프리셋 3종 등록 (신규 도구 아님) |

### 5.6 핵심 구현 사항

**session.js — 세션 컨텍스트:**

```javascript
// MCP Server 프로세스 내 싱글톤 인메모리 상태
// stdio 단일 프로세스이므로 동시성 문제 없음
const session = {
  lastEntryId: null,
  lastProjectId: null,

  setLastEntry(entryId, projectId) {
    this.lastEntryId = entryId;
    this.lastProjectId = projectId;
  },

  getLastEntryId() {
    if (!this.lastEntryId) throw new Error('세션 내 이전 분석 결과가 없습니다. 먼저 프롬프트를 분석해주세요.');
    return this.lastEntryId;
  }
};
```

**tagger.js — 자동 태그 추천 (local 모드):**

```javascript
// 키워드 → 태그 매핑 테이블. 별도 파일로 분리하여 확장 용이하게 유지.
const TAG_RULES = [
  { keywords: ['번역', 'translate', '영어로', '한국어로'], tag: '#translate' },
  { keywords: ['요약', 'summarize', '정리해줘'], tag: '#summarize' },
  { keywords: ['코드', 'code', '함수', 'function', '구현'], tag: '#coding' },
  { keywords: ['json', 'yaml', 'xml', '표', 'table'], tag: '#structured' },
  { keywords: ['분석', 'analyze', '비교'], tag: '#analysis' },
  // ...
];

function suggestTags(promptText) {
  const lower = promptText.toLowerCase();
  return TAG_RULES
    .filter(rule => rule.keywords.some(kw => lower.includes(kw)))
    .map(rule => rule.tag)
    .slice(0, 5); // 최대 5개
}
```

**improver.js — 자동 개선 루프 (무한 루프 방지):**

```javascript
async function runImprovementLoop(startEntryId, { targetScore = 85, maxIterations = 5 } = {}) {
  let currentId = startEntryId;
  const history = [];

  for (let i = 0; i < maxIterations; i++) {
    const entry = await storage.getEntry(currentId);
    history.push({ iteration: i + 1, entryId: currentId, score: entry.score });

    if (entry.score >= targetScore) {
      return { goalReached: true, iterations: i + 1, finalId: currentId, history };
    }

    // 개선 생성 → 재분석 → 새 entryId
    const newId = await generateAndSaveImproved(currentId);
    currentId = newId;
  }

  // maxIterations 초과: 현재까지 최고 점수 반환, 에러 아님
  return { goalReached: false, iterations: maxIterations, finalId: currentId, history };
}
```

## 5.7 `>> last N` 구현 계획 (v0.5.2)

### 변경 파일

| 파일 | 변경 내용 |
|------|----------|
| `lib/session.js` | `recentEntryIds` 링 버퍼 추가, `pushRecentEntryId`, `getRecentEntryIdAt`, `getRecentEntryCount` 함수 추가 |
| `lib/handlers/analysis.js` | `>> last N` 트리거 파싱, `setLastEntryId` → `pushRecentEntryId` 교체, `>> re` 로직 통일 |

### session.js 변경

```javascript
const sessionCtx = {
  lastEntryId: null,      // 기존 (>> re 하위 호환용)
  lastProjectId: null,
  recentEntryIds: [],     // 신규: 최대 10개 링 버퍼 (인덱스 0 = 가장 최근)
};

// 분석 완료 시마다 호출. lastEntryId도 함께 갱신하여 기존 코드 호환 유지.
export function pushRecentEntryId(id) {
  sessionCtx.recentEntryIds.unshift(id);
  if (sessionCtx.recentEntryIds.length > 10) {
    sessionCtx.recentEntryIds.pop();
  }
  sessionCtx.lastEntryId = id;
}

// n: 1-based (1 = 가장 최근)
export function getRecentEntryIdAt(n) {
  return sessionCtx.recentEntryIds[n - 1] ?? null;
}

export function getRecentEntryCount() {
  return sessionCtx.recentEntryIds.length;
}
```

### analysis.js 트리거 변경

```javascript
// 기존 >> re / >> v2 처리를 getRecentEntryIdAt(1)로 통일
const isReTrigger = /^>>\s*(re|재분석|v2)$/i.test(trimmed);
if (isReTrigger) {
  const lastId = getRecentEntryIdAt(1);
  if (!lastId) return '⚠️ 세션 내 이전 entryId가 없습니다.';
  return `직전 entryId \`${lastId}\` 를 parentId로 설정했습니다.\n새 프롬프트를 입력해 주세요.`;
}

// >> last N 신규 처리
const lastMatch = /^>>\s*last\s+(\d+)$/i.exec(trimmed);
if (lastMatch) {
  const n = parseInt(lastMatch[1], 10);
  if (n < 1 || n > 10) return `⚠️ N은 1~10 사이여야 합니다. (입력값: ${n})`;
  const targetId = getRecentEntryIdAt(n);
  const count = getRecentEntryCount();
  if (!targetId) {
    return `⚠️ 세션 내 ${n}번째 이전 분석이 없습니다. (현재 ${count}건 기록됨)`;
  }
  return [
    `${n}번째 이전 entryId \`${targetId}\` 를 parentId로 설정했습니다.`,
    `새 프롬프트를 입력해 주세요 (parentId: \`${targetId}\` 로 자동 연결됩니다).`,
  ].join('\n');
}
```

### setLastEntryId → pushRecentEntryId 교체

`analyzePrompt()` 내 `setLastEntryId(entryId)` 호출을 `pushRecentEntryId(entryId)`로 교체한다. `pushRecentEntryId`가 `lastEntryId`도 함께 갱신하므로 기존 `getLastEntryId()` 호출 코드는 수정 불필요.
