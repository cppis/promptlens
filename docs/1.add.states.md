## 2. 5축 분석 구조

`analyze_prompt` (local 모드)가 산출하는 5개 축. 각 0~100 점수, 평균이 최종 점수.

| 축 | 측정 기준 |
|----|----------|
| **Clarity** | 문장 구조, 길이, 제약 조건 명시 여부 |
| **Specificity** | 구체적 수치/이름/예시 포함 여부 |
| **Context** | 배경 정보, 역할(Role) 명시 여부 |
| **Structure** | 출력 형식, 문장 수, 줄바꿈 여부 |
| **Actionability** | Role + Format + Context + Example + Constraint 조합 |

등급 기준: A (90+) / B (70+) / C (50+) / D (50 미만)

## 3. 컨텍스트 상태 변환 다이어그램

### 3.1 프로젝트 생명주기

```
┌──────────┐  create_project  ┌────────────────┐  set_active_project  ┌──────────────────────┐
│  (없음)   │ ──────────────→  │  프로젝트       │  (action: "set")     │  활성 프로젝트         │
│          │                  │  (비활성)       │ ──────────────────→  │  (auto-save ON)       │
└──────────┘                  └────────────────┘ ←──────────────────  └──────────────────────┘
                                      │           set_active_project           │
                                      │           (action: "clear")            │
                                      │                                        │
                                      │◄── snapshot_project ───────────────────┤
                                      │◄── export_project                      │
                                      │◄── visualize_project                   │
                                      │◄── import_project                      │
                                      └──── (엔트리 축적 공간)
```

### 3.2 프롬프트 엔트리 생명주기

```
  사용자가 프롬프트 작성
          │
          ▼
  analyze_prompt (v1)
  ├── entryId 발급
  ├── 5축 점수 산출 (0~100)
  ├── 등급 부여 (A/B/C/D)
  ├── missingElements 식별
  └── 활성 프로젝트에 자동 저장
          │
    ┌─────┼────────────┐
    ▼     ▼            ▼
get_    get_         사용자가
history versions     프롬프트 수정/개선
(조회)  (v1만 존재)        │
                          ▼
                  analyze_prompt (v2, parentId = v1)
                  ├── 새 entryId 발급
                  ├── v1과 버전 체인 연결
                  └── 점수 변화 추적
                          │
              ┌───────────┼────────────┐
              ▼           ▼            ▼
       compare_       get_         추가 개선 반복
       prompts        versions      (v3, v4 ...)
       (v1, v2)       (v1→v2 체인)
```

### 3.3 분석 모드

`analyze_prompt`의 `mode` 파라미터로 선택. `deep`은 별도 모드가 아닌 `>> deep` 커맨드 트리거 시 실행되는 파이프라인 패턴.

```
               analyze_prompt
                     │
          ┌──────────┴──────────┐
          ▼                     ▼
    local 모드              api 모드
    (기본값, 무료)           (API 키 필요)
                                 │
                          set_api_key 선행 필요
          │                      │
          ▼                      ▼
  규칙 엔진 5축 스코어링    Claude API 3색 리포트
  즉시 반환                Referenced / Inferred / Missing
          │                      │
          └──────────┬───────────┘
                     ▼
             entryId + 분석 결과 반환
```

### 3.4 핵심 워크플로우: 프롬프트 개선 루프 (`>> deep`)

```
  ① 초기 작성       ② 분석              ③ 개선           ④ 재분석             ⑤ 비교
 ┌──────────┐  ┌──────────────┐   ┌──────────┐   ┌──────────────┐   ┌──────────────┐
 │ 프롬프트  │→ │analyze_prompt│→  │ 피드백    │→  │analyze_prompt│→  │compare_      │
 │ 작성     │  │ score: 45    │   │ 기반 수정 │   │ parentId=v1  │   │prompts(v1,v2)│
 └──────────┘  │ grade: D     │   └──────────┘   │ score: 82    │   │ Δ: +37       │
               │ missing: 3   │                  │ grade: A     │   └──────────────┘
               └──────────────┘                  │ missing: 0   │         │
                                                 └──────────────┘         ▼
                                                                    get_versions
                                                                     v1 → v2
```

## 4. 전제 조건 요약

| 명령어 | 전제 조건 |
|--------|----------|
| `analyze_prompt` | `prompt` 필수. `api` 모드는 API 키 필요. `parentId` 지정 시 해당 엔트리 존재 필요 |
| `compare_prompts` | `entryIdA`, `entryIdB` 모두 존재해야 함 |
| `get_versions` | 해당 `entryId` 존재 필요 |
| `set_active_project` (set) | 지정 프로젝트가 이미 생성되어 있어야 함 |
| `export_project` | 대상 프로젝트에 1개 이상의 엔트리 존재 필요 |
| `import_project` | 유효한 `.promptic.json` 파일 필요 |
| `set_api_key` | 유효한 Anthropic API 키 필요 |
| `snapshot_project` | `compareWith` 사용 시 비교 대상 스냅샷 존재 필요 |

## 5. 트리거 커맨드 패턴

### 5.1 `>> 접미사` 커맨드

| 커맨드 (EN) | 커맨드 (KR) | 동작 |
|-------------|-------------|------|
| `>> anz` | `>> 분석` | `analyze_prompt` local 모드 실행 |
| `>> anz+run` | `>> 분석+실행` | 분석 후 `enhancedPrompt` 즉시 실행 |
| `>> deep` | `>> 정밀분석` | 4단계 파이프라인: 분석 → 개선 → 재분석 → 비교 |
| `>> viz` | `>> 시각화` | `visualize_project` 실행 |
| `>> diff` | — | 최근 두 엔트리 `compare_prompts` |

### 5.2 자연어 트리거 (Claude가 자동 감지)

| 언어 | 예시 표현 |
|------|----------|
| EN | "analyze this prompt", "check this prompt", "how good is this prompt", "what's wrong with this", "how can I improve this" |
| KR | "이 프롬프트 분석해줘", "프롬프트 점검해줘", "이거 괜찮아?", "이 프롬프트 어때", "프롬프트 좀 봐줘", "이거 어떻게 개선해?" |
