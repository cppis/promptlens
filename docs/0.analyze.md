## 1. 분석 개요

PromptCraft의 분석은 "사용자의 프롬프트를 AI가 어떻게 읽는지"를 명시적으로 출력하는 것이다. Claude 같은 LLM은 프롬프트를 처리할 때 내부적으로 이미 용어 식별, 맥락 추론, 정보 부재 판단을 하고 있다. 다만 평소에는 그 과정을 사용자에게 보여주지 않을 뿐이다.

PromptCraft는 이 암묵적 과정을 명시적 JSON 출력으로 바꾸는 "메타 분석 레이어"다.

```
[사용자 프롬프트]
    │
    ▼
[PromptCraft 메타 분석 레이어]
    │  Claude API에 "이 프롬프트를 분석해달라"는
    │  별도의 시스템 프롬프트와 함께 전송
    │
    ▼
[Claude API 응답] → JSON { referenced, inferred, missing }
    │
    ▼
[클라이언트] 3색 리포트 렌더링 + 프로필 매칭 + 보강 프롬프트 생성
```

핵심 포인트: PromptCraft가 직접 NLP를 하는 것이 아니라, Claude에게 "너의 해석 과정을 구조화해서 알려달라"고 요청하는 구조다.

## 2. 분류 메커니즘

### 2.1 3색 분류 정의

| 분류 | 색상 | 의미 | AI 관점에서의 상태 |
|------|------|------|-------------------|
| Referenced | 초록 | 프롬프트에서 명시적으로 언급된 용어·개념 | "이건 확실히 안다" |
| Inferred | 오렌지 | 명시되지 않았지만 AI가 추론한 맥락 | "아마 이런 뜻일 것이다" |
| Missing | 빨강 | AI도 모르고, 없으면 답변 품질이 떨어지는 정보 | "이건 모르겠다, 알려달라" |

### 2.2 분류 기준 상세

**Referenced (참조됨):**

AI가 프롬프트 텍스트에서 직접 식별한 구체적 용어와 개념이다. 모호하지 않고 AI의 학습 데이터에 명확한 의미가 있는 것들이다.

```
프롬프트: "React로 로그인 폼 만들어줘"

Referenced:
  - "React" → JavaScript UI 라이브러리로 해석
  - "로그인 폼" → 사용자 인증 입력 UI로 해석
```

판단 기준: 프롬프트에 문자열로 존재하고, AI가 해당 용어의 의미를 특정할 수 있는 경우.

**Inferred (추론됨):**

프롬프트에 직접 언급되지 않았지만, Referenced 항목들과 일반적인 맥락에서 AI가 유추한 것이다. 각 항목에는 확신도(confidence)가 붙는다.

```
프롬프트: "React로 로그인 폼 만들어줘"

Inferred:
  - "웹 애플리케이션" → React 언급으로 웹 환경 추론 (확신: high)
  - "함수형 컴포넌트" → 최신 React 관행 추론 (확신: medium)
  - "비밀번호 암호화" → 로그인 기능의 보안 요구사항 추론 (확신: high)
```

확신도 기준:
- **high**: 거의 확실한 추론 (React → 웹 환경)
- **medium**: 그럴 가능성이 높은 추론 (React → 함수형 컴포넌트)
- **low**: 가능하지만 불확실한 추론 (Node.js → SSR)

판단 기준: 프롬프트에 문자열로 없지만, Referenced 항목으로부터 논리적으로 도출 가능한 경우.

**Missing (부재):**

AI가 답변을 생성할 때 필요하지만 프롬프트에도 없고 추론으로도 확정할 수 없는 정보다. 이것이 부재하면 AI는 임의로 가정하거나, 일반적인 답변으로 대체하게 된다.

```
프롬프트: "React로 로그인 폼 만들어줘"

Missing:
  - "인증 방식" → 이메일/비밀번호? OAuth? SSO?
  - "스타일링" → CSS Modules? Tailwind? styled-components?
  - "상태 관리" → useState? React Hook Form? Formik?
```

판단 기준: AI가 답변하려면 알아야 하는데, 프롬프트에 없고 추론만으로는 확정할 수 없는 경우.

### 2.3 분류 간 경계

세 분류는 명확히 나뉘는 것이 아니라 스펙트럼에 가깝다.

```
[명시적] ──────────────────────────────────────── [암묵적]
   │                    │                              │
 Referenced          Inferred                       Missing
 프롬프트에 있음    프롬프트에 없지만 추론 가능     추론도 불가
 확실히 알고 있음   아마 이럴 것이라 가정           모름/모호함
```

Claude API에게 이 스펙트럼 위에서 각 정보의 위치를 판단하게 하는 것이 시스템 프롬프트의 역할이다. 경계가 애매한 경우(예: "폼"이라는 단어가 있는데 "입력 검증"은 Referenced인가 Inferred인가?)에는 Claude의 판단에 맡기되, confidence 필드로 확신도를 표현한다.

## 3. Context 획득 방식

Context(맥락)는 두 가지 독립적인 소스에서 획득된다.

### 3.1 소스 1 — 사용자 프로필 (명시적)

사용자가 PromptCraft의 프로필 탭에서 직접 생성·관리하는 맥락이다.

```
ContextProfile {
  name: "웹 개발 프로젝트"
  role: "시니어 프론트엔드 개발자"
  domain: "에듀테크"
  techStack: ["React", "TypeScript", "Tailwind", "Node.js", "PostgreSQL"]
  preferences: "코드는 TypeScript, 설명은 한국어, 간결하게"
  customContext: "pnpm monorepo 구조 사용"
}
```

이 프로필은 분석 요청 시 `context` 파라미터로 함께 전송된다. Claude는 프롬프트와 이 맥락을 동시에 보고 분석한다. 맥락이 있으면 Missing 항목이 줄어든다 — "스타일링 방식"이 Missing일 때 프로필에 "Tailwind"가 있으면 Claude가 이를 감안해서 분석할 수 있다.

**프로필이 분석에 미치는 영향:**

```
프로필 없이 분석:
  Missing: 스타일링 방식, 상태 관리, 배포 환경, 대상 사용자 ... (8건)

프로필 있을 때 분석:
  Missing: 인증 방식, 비밀번호 정책, 라우팅 ... (5건)
  → 스타일링(Tailwind), 대상 사용자(에듀테크), 배포 환경 등은 프로필에서 보충됨
```

### 3.2 소스 2 — Missing → 프로필 자동 매칭 (자동)

Claude가 Missing으로 분류한 항목을 PromptCraft의 클라이언트 로직이 활성 프로필과 대조하여 자동으로 보충 가능한 항목을 식별한다.

이것은 Claude API 호출 이후, 클라이언트 측에서 수행하는 후처리다.

```
Claude 분석 결과에서 Missing: "스타일링 방식"
    │
    ▼
[클라이언트 매칭 로직]
    활성 프로필의 techStack 탐색
    → ["React", "TypeScript", "Tailwind", "Node.js", "PostgreSQL"]
    → CSS 프레임워크 카테고리에서 "Tailwind" 발견
    │
    ▼
결과: { field: "스타일링 방식", autoFillable: true, profileValue: "Tailwind" }
```

**매칭 테이블:**

| Missing 필드 카테고리 | 프로필에서 탐색하는 곳 | 매칭 조건 |
|--------------------|-------------------|---------|
| 스타일링/CSS 관련 | techStack | Tailwind, SCSS, styled-components, CSS Modules 등 |
| 상태 관리 | techStack | Redux, Zustand, Recoil, MobX 등 |
| 라우팅 | techStack | React Router, Next.js, Nuxt 등 |
| 데이터베이스 | techStack | PostgreSQL, MySQL, MongoDB, Redis 등 |
| 대상 사용자 | domain | domain 값 → "{domain} 분야 사용자" |
| 배포 환경 | techStack | AWS, Vercel, GCP, Azure 등 |
| 언어/톤 | preferences | 선호 스타일에서 추출 |

### 3.3 두 소스의 타이밍

```
[요청 시점]
    │
    ├── 소스 1: 활성 프로필을 context로 첨부 → Claude API에 전송
    │                                           (Claude가 맥락을 참고하여 분석)
    │
    ▼
[응답 수신 후]
    │
    ├── 소스 2: Missing 항목 vs 프로필 자동 매칭 → autoFillable 플래그 부여
    │                                              (클라이언트 후처리)
    │
    ▼
[보강 프롬프트 생성]
    = [맥락 블록] + [원본 프롬프트] + [자동 보충 항목]
```

소스 1은 분석 품질 자체를 높이고(Claude에게 더 많은 정보를 줌), 소스 2는 분석 결과에서 사용자 행동을 줄이는 역할(자동 보충)을 한다.

## 4. 프로토타입 vs 프로덕션 비교

| 항목 | 프로토타입 (현재) | 프로덕션 (계획) |
|------|----------------|---------------|
| 분류 엔진 | 규칙 기반 (키워드 매칭) | Claude API 메타 분석 |
| Referenced | KNOWN_TERMS 딕셔너리에서 문자열 검색 | Claude가 프롬프트에서 직접 식별 |
| Inferred | INFERENCE_RULES if-then 규칙 | Claude가 추론 과정을 명시적으로 출력 |
| Missing | MISSING_INFO_RULES 사전 정의 규칙 | Claude가 답변에 필요한 부재 정보 판단 |
| 커버리지 | 등록된 키워드만 (수십 개) | 모든 프롬프트 (Claude 학습 범위) |
| 정확도 | 낮음 (패턴 매칭 한계) | 높음 (LLM의 메타인지 활용) |
| 맥락 이해 | 없음 (단순 문자열) | 있음 (문맥, 뉘앙스, 도메인 이해) |
| 비용 | 0 (API 호출 없음) | Claude API 토큰 비용 발생 |
| 지연 | ~500ms (목 딜레이) | 1~3초 (Claude API 응답 시간) |

### 4.1 프로토타입 구현 (규칙 기반)

```javascript
// 키워드 딕셔너리에서 Referenced 추출
const KNOWN_TERMS = {
  'react': { explanation: 'JavaScript UI 라이브러리로 해석' },
  '로그인': { explanation: '사용자 인증 기능으로 해석' },
  ...
};

// if-then 규칙으로 Inferred 추론
const INFERENCE_RULES = [
  { keywords: ['react'], infer: { term: '웹 애플리케이션', confidence: 'high' }},
  { keywords: ['로그인'], infer: { term: '비밀번호 암호화 필요', confidence: 'high' }},
  ...
];

// 키워드 매칭으로 Missing 생성
const MISSING_INFO_RULES = [
  { keywords: ['react'], missing: [{ field: '스타일링 방식', suggestion: '...' }]},
  ...
];
```

한계: 등록되지 않은 키워드는 감지 불가. "Python으로 FastAPI 서버를 만들고 JWT 인증을 추가해줘" 같은 프롬프트에서 "FastAPI"가 KNOWN_TERMS에 없으면 Referenced에서 누락된다.

### 4.2 프로덕션 구현 (Claude API)

```javascript
// Claude API에 메타 분석 요청
const response = await claude.messages.create({
  model: 'claude-sonnet-4-6',
  system: ANALYSIS_SYSTEM_PROMPT,
  messages: [{
    role: 'user',
    content: `다음 프롬프트를 분석하라:\n\n${userPrompt}\n\n${contextBlock}`
  }]
});

// Claude가 JSON으로 응답
const analysis = JSON.parse(response.content[0].text);
// → { referenced: [...], inferred: [...], missing: [...] }
```

장점: Claude는 "FastAPI"를 당연히 알고 있으므로 Referenced에 정확히 넣는다. 또한 "FastAPI + JWT"라는 조합에서 "데이터베이스 필요", "토큰 만료 정책" 같은 미묘한 Missing도 판단할 수 있다.

## 5. 시스템 프롬프트 설계

### 5.1 기본 분석 프롬프트

```
당신은 프롬프트 분석 전문가입니다.
사용자가 AI에게 보낼 프롬프트를 받으면, 그 프롬프트를 AI의 관점에서 분석하여
다음 3가지 카테고리로 분류하세요.

1. referenced (참조됨):
   프롬프트에서 명시적으로 언급된 용어와 개념.
   AI가 해당 단어를 보고 구체적 의미를 특정할 수 있는 것.
   - 각 항목: { "term": "용어", "explanation": "AI가 이 용어를 어떻게 이해했는지" }

2. inferred (추론됨):
   프롬프트에 직접 쓰여 있지 않지만, referenced 항목들과 일반적 맥락에서
   AI가 유추할 수밖에 없는 배경 정보.
   - 각 항목: { "term": "추론 내용", "explanation": "추론 근거", "confidence": "high|medium|low" }
   - confidence 기준:
     high = 거의 확실 (React → 웹 환경)
     medium = 그럴 가능성 높음 (React → 함수형 컴포넌트)
     low = 가능하지만 불확실

3. missing (부재):
   AI가 좋은 답변을 하려면 알아야 하지만, 프롬프트에도 없고 추론으로도
   확정할 수 없는 정보. 이것이 없으면 AI는 임의로 가정하거나 일반적 답변으로 대체함.
   - 각 항목: { "field": "부재 정보명", "suggestion": "사용자에게 보여줄 보충 가이드" }

반드시 유효한 JSON 형식으로만 응답하세요.
```

### 5.2 맥락 포함 시 프롬프트

프로필 맥락이 있을 때는 시스템 프롬프트에 추가 지시를 넣는다.

```
(위 기본 프롬프트에 추가)

사용자의 맥락 정보가 함께 제공됩니다.
이 맥락을 참고하여 분석하되, missing 항목에서 맥락으로 이미 알 수 있는 것은
제외하거나 "맥락에서 보충 가능"으로 표시하세요.
```

### 5.3 프롬프트 튜닝 포인트

분석 품질을 높이기 위해 반복 튜닝해야 할 요소들:

| 튜닝 포인트 | 문제 | 해결 방향 |
|------------|------|---------|
| Referenced 과다 | 관사, 조사 같은 불필요한 단어까지 포함 | "기술적·도메인적으로 의미 있는 용어만" 지시 추가 |
| Inferred 과다 | 너무 먼 추론까지 나열 | confidence 기준 강화, "답변에 직접 영향을 미치는 추론만" |
| Missing 과다 | 사소한 것까지 부재로 분류 | "없으면 답변 품질이 유의미하게 떨어지는 것만" |
| Missing 과소 | 중요한 부재 정보를 놓침 | 도메인별 체크리스트를 시스템 프롬프트에 포함 |
| JSON 파싱 실패 | Claude가 JSON 외 텍스트 포함 | response_format 또는 더 엄격한 포맷 지시 |

## 6. 자동 보충 매칭 로직

### 6.1 매칭 흐름

```
Claude 응답의 missing 배열
    │
    ▼
각 missing 항목에 대해:
    │
    ├── field 값으로 카테고리 분류
    │   (스타일링? 상태관리? DB? 배포? 사용자? ...)
    │
    ├── 해당 카테고리에 맞는 프로필 필드 탐색
    │   (techStack? domain? preferences? ...)
    │
    ├── 프로필 값이 카테고리에 매칭되는지 확인
    │   (techStack에 "Tailwind" 있음 → 스타일링 카테고리에 매칭)
    │
    ▼
매칭 성공: { autoFillable: true, profileValue: "Tailwind" }
매칭 실패: { autoFillable: false, profileValue: null }
```

### 6.2 카테고리 → 프로필 필드 매핑

```javascript
const CATEGORY_MAP = {

                 'emotion', 'less', 'bootstrap', 'chakra ui', 'material ui']
```

### 6.3 프로덕션에서의 개선

프로토타입에서는 위 매핑을 하드코딩하지만, 프로덕션에서는 Claude에게 매칭 판단도 위임할 수 있다.

```
시스템 프롬프트에 추가:

missing 항목을 분류할 때, 사용자의 맥락 정보에서 해당 정보를 추출할 수 있는지도 판단하세요.
추출 가능한 경우 "autoFillable": true와 "profileValue": "매칭된 값"을 포함하세요.
```

이렇게 하면 카테고리 매핑 테이블 없이도 Claude가 맥락적으로 판단한다. 예를 들어 프로필에 "MSA 아키텍처 선호"라고 적혀 있고 Missing에 "아키텍처 패턴"이 있으면, 하드코딩된 매핑으로는 매칭이 어렵지만 Claude는 연결할 수 있다.

## 7. 한계와 개선 방향

### 7.1 현재 한계

| 한계 | 설명 | 영향 |
|------|------|------|
| Claude의 자기 인식 한계 | Claude가 "자기가 뭘 모르는지"를 완벽히 아는 것은 아님 | Missing 항목 누락 가능 |
| 분류 일관성 | 같은 프롬프트를 여러 번 분석하면 결과가 조금씩 다를 수 있음 | UX 혼란 |
| 도메인 편향 | 기술 프롬프트에 비해 비기술 프롬프트(마케팅, 글쓰기)는 분석 품질이 낮을 수 있음 | 사용자 층 제한 |
| 비용 | 매 분석마다 Claude API 호출 발생 | 프리미엄 모델 필요 |

### 7.2 개선 방향

**단기 (MVP 내):**

- 분석 결과 캐싱: 동일/유사 프롬프트에 대해 캐시된 결과 반환
- 시스템 프롬프트 반복 튜닝: 50개 테스트 프롬프트로 정확도 80% 목표
- 분석 결과에 대한 사용자 피드백 ("이 분류가 맞나요?") 수집

**중기 (Phase 2):**

- 도메인별 시스템 프롬프트: 개발, 마케팅, 글쓰기 등 도메인에 따라 분석 프롬프트 전환
- 분석 히스토리: 같은 사용자의 이전 분석 패턴을 참고하여 Missing 정확도 향상
- Claude의 매칭 위임: 자동 보충 매칭을 Claude에게도 수행하게 하여 하드코딩 의존도 감소

**장기 (Phase 3+):**

- 파인튜닝 또는 경량 모델: 분석 전용 경량 모델로 비용 절감
- 오프라인 분석: 자주 나오는 패턴을 로컬 규칙으로 변환하여 API 없이도 기본 분석 가능
- 사용자 피드백 루프: 사용자가 분류를 수정하면 해당 데이터로 분석 품질 개선
