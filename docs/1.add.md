## 1. 제품 설계

### 1.1 컨셉

> "AI가 내 말을 어떻게 이해했는지 보여주고, 부족한 맥락을 알아서 채워준다."

**제품 방향:** "교육"이 아니라 "즉시 실용" — AI 답변이 왜 이상한지 바로 보여주고, 다음에 뭘 보충하면 되는지 안내하며, 반복되는 약점은 자동으로 코칭해주는 MCP 도구.

**3가지 핵심 축:**

| 축 | 기능 | 사용자 가치 |
|---|------|----------|
| AI 해석 투명화 | 프롬프트를 넣으면 AI가 뭘 이해했고 뭘 모르는지 보여줌 | "AI 답변이 왜 이상한지" 즉시 파악 |
| AI 해석 시뮬레이션 | AI가 1인칭으로 "나는 이렇게 이해했고 여기서 추측했다"를 서술 | 점수가 아니라 AI의 사고 과정을 직접 체험 |
| 누적 패턴 코칭 | 분석 이력에서 반복 약점 추출 → 개인화 코칭 + 행동 제안 | 일회성 분석 → 지속적 성장 도구 |

**보조 기능 (확장 로드맵):**

| 기능 | 설명 |
|------|------|
| 맥락 빌더 | 프로젝트·역할·선호 스타일을 프로필로 저장, 필요한 맥락만 선별·삽입 |
| 프롬프트 변이 비교 (Prompt Lab) | 같은 의도의 프롬프트 2~3개를 변형 생성, AI 해석 차이를 나란히 비교 |
| 응답 품질 감사 (Response Auditor) | AI 답변의 환각, 논리 비약, 출처 불명확 주장 자동 탐지 |

**기술 스택:**

| 계층 | 기술 | 상태 |
|------|------|------|
| 클라이언트 | MCP Server (Node.js + @modelcontextprotocol/sdk, v0.4.0) | 확정 (현재) |
| AI 연동 | Claude API 직접 호출 (BYOK, api 모드) | 확정 |
| 분석 방식 | local 모드 (로컬 규칙, 무료), api 모드 (Claude API, BYOK) | 확정 |
| 저장소 | 프로젝트별 JSON 파일 (~/.promptic/projects/{id}.json) | 확정 (v0.4.0) |
| 비용 모델 | 사용자 본인 API 키 사용 (api 모드만) | 확정 |

### 1.2 핵심 철학 — AI 활용 역량 5가지

사용자가 AI를 더 잘 쓰려면 5가지 역량이 필요하고, Promptic는 도구 사용 과정에서 이를 자연스럽게 키워준다.

| 역량 | 정의 | Promptic가 돕는 방식 |
|------|------|---------------------|
| AI 해석 이해 | AI가 내 말을 어떻게 읽는지 아는 것 | 해석 투명화 + 해석 시뮬레이션 |
| 맥락 관리 | AI에게 필요한 배경 정보를 효율적으로 전달 | 맥락 빌더 (향후) |
| 결과 검증 | AI 답변의 품질과 신뢰도를 판단 | Response Auditor (향후) |
| 실험적 접근 | 다양한 표현으로 시도하고 비교하는 습관 | Prompt Lab (향후) |
| 패턴 학습 | 자신의 프롬프팅 습관을 인식하고 개선 | 누적 패턴 코칭 |

### 1.3 기능 메카닉 상세

#### 1.3.1 AI 해석 투명화 (Interpretation Transparency)

**목적:** 사용자의 프롬프트를 AI가 어떻게 해석했는지 분해하여, "AI 답변이 왜 이상한지"를 즉시 파악할 수 있게 한다.

**출력 — 3색 해석 리포트:**

| 영역 | 의미 | 예시 |
|------|------|------|
| Referenced (초록) | AI가 프롬프트에서 명확히 식별한 용어·개념 | "React" → JavaScript UI 라이브러리로 해석 |
| Inferred (오렌지) | AI가 직접 받지 않았지만 추론한 배경 맥락 | "웹 애플리케이션" → React 언급에서 추론 (확신: 높음) |
| Missing (빨강) | AI가 모르는 정보, 모호하거나 부재한 것 | "인증 방식" → 이메일/비밀번호? OAuth? SSO? |

**동작 흐름:**

```
사용자 프롬프트 입력 (Claude Desktop/Code에서 자연어로 요청)
    │
    ▼
[MCP Server] analyze_prompt 도구 호출
    │
    ├── local 모드: 규칙 엔진 → 5축 점수 + 누락 요소 + 개선 제안 (무료, 즉시)
    ├── api 모드: Claude API → 3색 리포트 (BYOK, 정밀)
    │
    ▼
[응답] 점수, 누락 요소, 제안, 보강된 프롬프트 반환
```

**분석용 시스템 프롬프트 (api 모드):**

```
사용자의 프롬프트를 분석하여 다음 3가지 카테고리로 분류하라:
1. referenced: 프롬프트에서 명확히 이해한 용어와 개념 (각 항목에 term, explanation)
2. inferred: 명시되지 않았지만 추론한 맥락 (각 항목에 term, explanation, confidence)
3. missing: 부재하거나 모호한 정보 (각 항목에 field, suggestion)
JSON 형식으로 응답하라.
```

**API 응답 스키마:**

```json
{
  "referenced": [{ "term": "...", "explanation": "..." }],
  "inferred": [{ "term": "...", "explanation": "...", "confidence": "high|medium|low" }],
  "missing": [{ "field": "...", "suggestion": "..." }],
  "enhancedPrompt": "..."
}
```

#### 1.3.2 AI 해석 시뮬레이션 (Interpretation Narrative) — v0.5.0

**목적:** 3색 리포트의 분류 목록을 넘어, AI가 1인칭 시점으로 "이 프롬프트를 읽고 어떻게 이해했는지"를 자연어로 서술한다.

**핵심 가치:** 프롬프팅 지식이 없는 일반 사용자도 "아, AI가 여기서 이렇게 추론하는구나"를 즉시 파악. 진입 장벽이 가장 낮고 임팩트가 가장 큰 기능.

**기존 분석과의 차이:**

| 항목 | 기존 (3색 리포트) | 해석 시뮬레이션 |
|------|------------------|---------------|
| 형태 | 분류 목록 | AI 1인칭 서술문 |
| 알려주는 것 | "무엇이 부족한지" | "AI가 어떻게 오해하는지" |
| 필요 지식 | 5축 점수 기준 이해 필요 | 없음 (자연어 읽기만 하면 됨) |

**출력 형식:**

```
[AI의 시점]

나는 이 프롬프트를 받고 당신이 REST API 설계를 원한다고 이해했다.

"시니어 백엔드 개발자"라는 역할 지정이 있어서, 전문적인 수준의 아키텍처 패턴과
실무 용어를 사용하여 답변하겠다.

다만 몇 가지를 직접 추측해야 했다:
- 시스템 규모를 명시하지 않아서 중소규모로 가정했다.
- "좋은 설계"의 기준이 없어서 일반적인 베스트 프랙티스를 나열할 것이다.

출력 형식을 지정하지 않았으므로 자유 서술로 답하겠다.
```

**구현 방식:**

| 모드 | 방식 | 비용 |
|------|------|------|
| api 모드 | Claude API 시스템 프롬프트에 서술형 응답 지시 추가 | API 호출 비용 |
| local 모드 | 규칙 엔진 분석 결과를 템플릿 기반 서술문으로 변환 | 무료 |

**MCP 도구 변경:** `analyze_prompt` 응답에 `narrative` 필드 추가. HistoryEntry에도 `narrative` 저장.

#### 1.3.3 누적 패턴 코칭 (Personal Pattern Coach) — v0.5.0

**목적:** 축적된 히스토리에서 반복 약점 패턴을 추출하고, 개인화된 코칭과 구체적 행동 제안을 제공한다.

**기존 통계와의 차이:**

| 항목 | 기존 (get_stats) | 누적 패턴 코칭 |
|------|-----------------|---------------|
| 데이터 | 평균 점수, 태그 빈도 | 축별 약점 패턴, 도메인별 분석, 성장 추이 |
| 인사이트 | "평균 62점" | "코드 리뷰에서 Format 누락률 78%" |
| 행동 제안 | 없음 | "다음 프롬프트에 '제한사항: ...'을 추가해보세요" |

**출력 — 코칭 리포트:**

```
📊 이번 주 요약
- 분석한 프롬프트: 23개, 평균 점수: 68점 (+6점 ↑)

🔍 반복 패턴
1. 출력 형식 누락 (78%) → 18/23개에서 format 미지정
2. 제약 조건 미명시 (65%)

3. 도메인별 차이
   → 코드 리뷰: Context 82점 (강점), Format 35점 (약점)
   → 데이터 분석: Role 88점 (강점), Example 22점 (약점)

💡 코칭
- "출력 형식: 표 형태로, 열은 [항목, 설명, 우선순위]"를 추가해보세요

📈 성장 추이: W1 52 → W2 58 → W3 62 → W4 68 (+31%)
```

**분석 로직 (local, API 불필요):**

```
get_coaching 호출
    ├── 히스토리 수집 → 축별 통계 → Missing 패턴 빈도
    ├── 도메인별 교차분석 → 성장 추이 계산
    └── 코칭 메시지 생성 (템플릿 기반)
```

**MCP 도구:** `get_coaching` (파라미터: period, projectId). 최소 10개 히스토리 필요.

#### 1.3.4 프롬프트 변이 비교 — Prompt Lab (이후 버전)

같은 의도의 프롬프트를 2~3개 변형하여, "어떤 표현이 AI 해석에 어떤 차이를 만드는지" 비교. 사람은 추상적 조언보다 구체적 비교에서 훨씬 빠르게 학습한다.

#### 1.3.5 응답 품질 감사 — Response Auditor (이후 버전)

AI 답변의 환각, 논리 비약, 출처 불명확 주장을 자동 탐지. 각 마킹 항목에 "이것을 확인하려면..." 후속 질문 제안.

### 1.4 수익 모델

> **현재 (BYOK):** 사용자가 자신의 Claude API 키를 사용하므로 운영 비용 0. 수익화는 이후 버전에서 검토.

| 방식 | 설명 |
|------|------|
| 무료 오픈소스 | BYOK 전용 완전 무료, 커뮤니티 기반 성장 |
| 프리미엄 기능 과금 | Prompt Lab, Response Auditor 등 고급 기능만 유료 |

## 2. 시장 조사

### 2.1 시장 분류

#### 1유형 — 원클릭 프롬프트 개선기 (가장 많음)

| 제품 | 형태 | 핵심 기능 |
|------|------|---------|
| Promptimize | 크롬 확장 | 프롬프트 자동 개선, 변수 템플릿 |
| Prompter | 크롬 확장 | 워크스페이스 컨텍스트 자동 반영 |
| Pretty Prompt | 크롬 확장 + 웹앱 | 원클릭 프롬프트 리파인 |
| Prompt Fixer | 크롬 확장 | 프롬프트 점수 + 개선 |
| PromptPerfect | 웹앱 | 프롬프트 생성 및 최적화 |

공통 한계: 사용자가 "왜" 개선되었는지 학습하지 못함.

#### 2유형 — 분석 + 부족 정보 탐지 (가장 유사)

| 제품 | 형태 | 핵심 기능 |
|------|------|---------|
| FunBlocks AI Prompt Optimizer | 크롬 확장 | 프롬프트 분석, 동적 폼 생성 |
| Prompt Analyzer (GPT) | ChatGPT GPT | LLM 프롬프트 구조/스타일/효과성 피드백 |

FunBlocks 한계: 프롬프트를 "고쳐주는 것"이지 "AI가 어떻게 읽었는지" 보여주는 것은 아님.

#### 3유형 — 점수/효율 측정기

| 제품 | 형태 | 핵심 기능 |
|------|------|---------|
| Promptlee | 크롬 확장 | 효율성 점수(0~100), 토큰 수/비용 추정 |

### 2.2 시장 공백

| 비교 항목 | 기존 툴 | Promptic |
|---------|---------|-------------|
| 핵심 행위 | 프롬프트를 자동으로 고쳐줌 | AI가 어떻게 읽었는지 보여줌 |
| 부족 정보 | 폼으로 채우게 함 | AI 관점에서 왜 부족한지 설명 |
| 시간 축 | 일회성 | 누적 패턴 코칭 |
| 사용자에게 남는 것 | 더 나은 텍스트 | AI 활용 역량 향상 |

**시장에 없는 기능:** AI 해석 시뮬레이션 (1인칭 서술), 누적 패턴 코칭 (개인화 행동 제안), MCP 네이티브 (대화 중 즉시 사용)

### 2.3 차별화 포지셔닝

```
 자동 개선 ←─────────────────────→ 역량 향상
 │                                           │
 Promptimize      FunBlocks      Promptic  │
 Pretty Prompt    Prompt         (목표 위치)  │
 Prompt Fixer     Analyzer                    │
 │                                            │
 └──── 확장 도구 ─────────── MCP 도구 ────────┘
```

Promptic는 **MCP 네이티브 도구**이면서 **AI 활용 역량 향상에 초점**을 맞추는 유일한 포지션.

## 3. 로컬 파일 저장소 기능 설계 (v0.4.0)

> Feature: **Project Local Export & Query** — 구현 완료 (2026-03-24)

### 3.1 문제 정의

#### 3.1.1 기존 저장소의 한계

v0.3.0까지 Promptic는 모든 데이터를 단일 파일(`~/.promptic/data.json`)에 저장했다. 이 구조는 프로젝트가 늘어날수록 세 가지 문제를 드러냈다.

**문제 1 — 파일 비대화:** 10개 프로젝트, 각 100개 이력이면 data.json 하나가 수백 KB가 된다. 전체를 읽고 쓰는 비용이 선형으로 증가한다.

**문제 2 — 공유 불가:** 특정 프로젝트의 이력을 팀원에게 보내거나 다른 기기로 옮기려면 전체 data.json을 복사해야 한다. 프로젝트 단위 이동/공유 수단이 없다.

**문제 3 — 쿼리 한계:** `get_history`의 `search` 파라미터는 단순 문자열 포함 여부만 확인한다. 점수 범위, 날짜 구간, 등급 필터, 버전 체인만 보기 등 실용적인 쿼리를 지원하지 않는다.

#### 3.1.2 해결하려는 것

- 프로젝트별 독립 파일로 저장 구조를 분리한다.
- JSON·Markdown·CSV 3가지 형식으로 내보내고 가져올 수 있게 한다.
- 점수·날짜·등급·태그·버전 기준의 고급 쿼리를 지원한다.
- 스냅샷으로 특정 시점의 프로젝트 상태를 보존한다.

### 3.2 사용자 스토리

**시나리오 A — 개인 백업 & 이식성**

> "지난 한 달간 'Backend API' 프로젝트에서 분석한 프롬프트들을 Markdown으로 내보내서 개인 노트에 붙여넣고 싶다."

```
"Backend API" 프로젝트를 Markdown으로 내보내줘
→ ~/.promptic/exports/backend-api-2026-03-24.md 생성
```

**시나리오 B — 팀 공유**

### 3.4 내보내기 형식

| 형식 | 용도 | 재가져오기 |
|------|------|-----------|
| `json` (`.promptic.json`) | 팀 공유, 다른 머신 복원, 완전한 데이터 보존 | ✅ (`import_project`로) |
| `markdown` (`.md`) | 개인 노트 앱, GitHub, Notion 붙여넣기, 주간 리포트 | ❌ |
| `csv` (`.csv`) | Excel/Google Sheets 분석, 점수 추이 시각화 | ❌ |

Markdown 리포트 구조:
```
# {project.name} — Promptic 분석 리포트
> 기간 | 총 N개 분석 | 평균 점수

## 요약
- 최고/최저 점수
- 가장 많이 누락된 요소 Top 3
- 등급 분포

## 분석 기록 (최신순)
### [{date}] {score}/100 ({grade}등급)
원본 / 누락 요소 / 개선안 / 태그
```

CSV는 `axisScores`를 5개 컬럼으로 플래트닝, `tags`·`missingElements`는 세미콜론 구분 문자열로 저장한다.

### 3.5 고급 쿼리 (`query_history`) 설계

| 파라미터 | 타입 | 설명 |
|----------|------|------|
| `scoreMin` / `scoreMax` | number | 점수 범위 필터 |
| `grade` | string[] | 등급 AND 필터 (`["A","B"]`) |
| `dateFrom` / `dateTo` | string | ISO 8601 날짜 범위 |
| `tags` | string[] | AND 조건 태그 배열 |
| `tagsAny` | string[] | OR 조건 태그 배열 |
| `missing` | string[] | 누락 요소 포함 항목만 |
| `versionsOnly` | boolean | 버전 체인 루트만 반환 |
| `search` | string | 프롬프트·노트·태그 전문 검색 |
| `sortBy` | string | `score_asc/desc`, `date_asc/desc` |

`get_history`는 단순 조회 용도로 유지(하위 호환)하고, `query_history`는 고급 필터가 필요할 때 사용한다.

### 3.6 스냅샷 비교 출력

```
📊 Backend API 스냅샷 비교 (2026-02-24 → 2026-03-24)

분석 수:     23 → 42 (+19)
평균 점수:   58 → 71 (+13점)
A등급 비율:  4% → 19% (+15%)
D등급 비율:  39% → 12% (−27%)

자주 누락된 요소 변화:
  output_format: 19회 → 9회 (▼53%)
  constraints:   15회 → 6회 (▼60%)
  role:           8회 → 2회 (▼75%)
```

### 3.7 제약 및 고려사항

**파일 크기 제한:** 단일 프로젝트 파일이 10MB를 초과하면 경고를 반환한다 (히스토리 500개 이상 시 해당 가능성).

**동시성:** MCP는 stdio 단일 프로세스이므로 동시 쓰기 충돌은 없다. 파일 쓰기는 원자적 교체(temp 파일 → rename)로 처리한다.

**CSV 한계:** CSV는 Import 지원 안 함 (Export 전용). Import는 JSON 형식만 지원한다.

**마이그레이션 안전성:** 마이그레이션 실패 시 기존 `data.json`이 그대로 유지되어 서버는 정상 동작한다. 성공 후에도 `data.json.migrated`를 보존한다.

**내보내기 경로 보안:** `outputPath` 파라미터는 `~/.promptic/` 하위로만 제한한다. 임의 경로 쓰기를 방지한다.

## 4. 사용자 편의성 강화 기능 설계 (v0.5.1)

> Feature: **UX Convenience Enhancements** — 사용자 편의성 기준 상위 5개 기능

### 4.1 개요

v0.4.0(저장소), v0.5.0(AI 시뮬레이션·코칭)이 분석 품질을 높였다면, v0.5.1은 **반복 마찰 제거**에 집중한다. 매 세션마다 발생하는 수동 작업을 자동화하여 프롬프트 개선 루프의 속도를 높인다.

| 기능 | 마찰 제거 대상 | 임팩트 |
|------|--------------|--------|
| 자동 개선 생성 (`>> fix`) | 분석 후 수동 수정 작업 | 최상 |
| 직전 entryId 자동 참조 (`>> re`) | 긴 ID 복사·붙여넣기 | 상 |
| 자동 태그 추천 | 수동 태그 입력 | 상 |
| 목표 점수 자동 개선 루프 (`>> loop`) | 개선→재분석→비교 수동 반복 | 상 |
| 배치 분석 (`batch_analyze`) | 프롬프트 하나씩 개별 호출 | 중 |
| MCP Prompts 프리셋 확장 | `/` 메뉴에서 도메인별 프리셋 없음 | 상 |
| 개선 제안 클립보드 포맷 출력 | enhancedPrompt가 JSON 안에 묻혀 복사 불편 | 상 |

### 4.2 자동 프롬프트 개선 생성 (`>> fix`)

**목적:** `analyze_prompt` 결과의 `missingElements`와 5축 점수를 바탕으로 Claude API가 개선된 프롬프트를 즉시 생성한다. 생성본은 자동으로 `parentId`가 연결된 새 엔트리로 저장된다.

**트리거:**

| 영어 | 한글 | 동작 |
|------|------|------|
| `>> fix` | `>> 개선` | 직전 분석 결과 기반 개선 버전 생성 |
| `>> fix+save` | `>> 개선+저장` | 개선 버전 생성 후 히스토리에 자동 저장 |

**개선 생성 원칙:**
- 원본 프롬프트의 핵심 의도를 유지한다
- `missingElements`에 명시된 요소만 보완한다 (불필요한 내용 추가 금지)
- 생성된 개선본은 `parentId: 원본 entryId`로 자동 연결된다

**출력 형식:**

```
원본 점수: 45점 (D등급) → 예상 개선 점수: 78점 (B등급)

[개선된 프롬프트]
당신은 시니어 백엔드 개발자입니다. 다음 REST API를 설계해주세요.
요구사항: ...
출력 형식: OpenAPI 3.0 YAML 형식으로, 엔드포인트별 request/response 스키마 포함
제약사항: Node.js Express 기반, 인증은 JWT 사용

변경 요약:
- 추가: 출력 형식 명시 (output_format 누락 보완)
- 추가: 제약사항 명시 (constraints 누락 보완)

entryId: `h_xxxxxxxxxx_xxxxxx` (>> fix+save 시)
```

**전제 조건:** api 모드 API 키 필요. 생략 시 local 모드로 규칙 기반 보완.

### 4.3 직전 entryId 자동 참조 (`>> re`)

**목적:** 세션 내 가장 최근 발급된 `entryId`를 자동으로 `parentId`로 사용한다. 긴 ID를 복사·붙여넣기하지 않아도 버전 체인을 이을 수 있다.

**트리거:**

| 영어 | 한글 | 동작 |
|------|------|------|
| `>> re` | `>> 재분석` | 직전 entryId를 parentId로 하여 새 프롬프트 분석 |
| `>> v2` | `>> v2` | 동일 (별칭) |

**동작 규칙:**
- 세션 컨텍스트에서 마지막으로 반환된 `entryId`를 자동 참조한다
- 명시적 `parentId`가 제공된 경우 자동 참조를 무시한다
- 세션 내 이전 `entryId`가 없으면 오류를 안내하고 일반 분석으로 폴백한다

**사용 예시:**

```
# 기존 방식 (불편)
>> 분석 (parentId: h_a1b2c3d4e5_f6g7h8)

# 개선 방식
>> re
→ 세션의 직전 entryId를 parentId로 자동 설정
```

**구현:** MCP Server 프로세스 내 인메모리 세션 상태(`lastEntryId`)로 관리. 서버 재시작 시 초기화되므로 영속성 저장 불필요.

### 4.4 자동 태그 추천

**목적:** `analyze_prompt` 실행 시 프롬프트 내용을 분석해 태그를 자동으로 추천한다. 수동 태그 입력 없이도 `query_history` 필터가 의미 있게 동작하도록 히스토리 분류 품질을 높인다.

**모드별 방식:**
- **local 모드**: 규칙 기반 키워드 감지로 태그 제안 (비용 없음)
- **api 모드**: Claude API 경량 호출로 의미 기반 태그 생성 (분석 호출과 병렬 실행)

**자동 감지 태그 분류:**

| 분류 | 예시 태그 |
|------|----------|
| 도메인 | `#coding`, `#writing`, `#analysis`, `#data`, `#creative` |
| 용도 | `#summarize`, `#translate`, `#explain`, `#generate`, `#review` |
| 출력 형식 | `#structured`, `#list`, `#json`, `#prose`, `#table` |
| 품질 힌트 | `#needs-context`, `#ambiguous`, `#well-defined` |

**`analyze_prompt` 파라미터 추가:**

| 파라미터 | 기본값 | 설명 |
|----------|--------|------|
| `autoTag` | `true` | 자동 태그 추천 활성화 여부 |
| `tagMode` | `suggest` | `suggest`(응답에만 표시) / `auto`(히스토리에 자동 저장) |

**응답 변경:** 기존 응답에 `suggestedTags` 필드 추가.

```json
{
  "suggestedTags": ["#coding", "#explain", "#structured"],
  "appliedTags": ["#coding", "#explain"]
}
```

### 4.5 목표 점수 자동 개선 루프 (`>> loop`)

**목적:** 목표 점수를 지정하면 "개선 생성 → 재분석 → 비교"를 자동 반복하고, 목표 달성 시 최종 버전과 전체 개선 체인 요약을 반환한다.

**트리거:**

| 영어 | 한글 | 동작 |
|------|------|------|
| `>> loop 90` | `>> 루프 90` | 목표 점수 90점 달성까지 자동 반복 |
| `>> loop` | `>> 루프` | 기본 목표 점수(85)로 자동 반복 |

**파라미터:**

| 파라미터 | 기본값 | 설명 |
|----------|--------|------|
| `targetScore` | `85` | 목표 종합 점수 (0~100) |
| `maxIterations` | `5` | 최대 반복 횟수 (무한 루프 방지) |

**동작 흐름:**

```
시작 entryId (세션 내 직전 ID 자동 참조)
    │
    ▼
[개선 생성 (>> fix 내부 로직)]
    │
    ▼
[재분석 (analyze_prompt, parentId 자동 연결)]
    │
    ├── 목표 달성? ──Yes──→ [최종 결과 반환]
    │
    └── No → maxIterations 초과? ──Yes──→ [중단 + 현재 최고점 반환]
                │
                └── No → 반복
```

**출력 형식:**

```
루프 완료: 3회 반복 (목표: 85점)

v1 (h_aaa): 45점 → v2 (h_bbb): 67점 → v3 (h_ccc): 88점 ✅

[최종 프롬프트 전문]
최종 entryId: `h_ccc_xxxxxx`
```

**전제 조건:** api 모드 API 키 필수. `>> fix` 내부 로직 구현 완료 후 활성화.

### 4.6 배치 분석 (`batch_analyze`)

**목적:** 여러 프롬프트를 한 번에 제출해 일괄 분석하고 비교 테이블을 반환한다. 새 프로젝트 시작 시 후보 프롬프트 평가 또는 기존 프롬프트 일괄 점수화에 유용하다.

**트리거:**

| 한글 명령 예시 | 동작 |
|--------------|------|
| "여러 프롬프트 한 번에 분석해줘" | `batch_analyze` 실행 |
| "후보 프롬프트들 비교해줘" | 배치 분석 후 순위 표 반환 |

**파라미터:**

| 파라미터 | 필수 | 설명 |
|----------|------|------|
| `prompts` | 필수 | 분석할 프롬프트 배열 (최대 20개) |
| `mode` | 선택 | `local` / `api` (기본값: `local`) |
| `projectId` | 선택 | 저장 대상 프로젝트 |
| `saveAll` | 선택 | 모든 결과 히스토리 저장 여부 (기본값: `false`) |
| `tags` | 선택 | 배치 전체 공통 적용 태그 |

**출력 형식:**

```
배치 분석 결과 (5개 프롬프트)

순위 | 점수 | 등급 | entryId        | 프롬프트 요약
  1  |  88  |  A   | h_aaa_111      | "당신은 전문 번역가..."
  2  |  74  |  B   | h_bbb_222      | "다음 텍스트를 번역해줘..."
  3  |  61  |  C   | h_ccc_333      | "번역해줘..."

최고 점수: h_aaa_111 (88점, A등급)
평균 점수: 74.3점
```

**제약 조건:**
- 단일 호출당 최대 20개 프롬프트
- api 모드는 요청 수만큼 API 비용 발생 안내
- 동시 API 요청 최대 3개로 rate limit 방어 (semaphore)
- 동일 배치 내 프롬프트는 자동으로 `#batch` 태그 추가

### 4.7 도구 변경 요약

| 구분 | 변경 내용 |
|------|----------|
| 신규 도구 | `batch_analyze` |
| 신규 트리거 | `>> fix`, `>> fix+save`, `>> re`, `>> v2`, `>> loop`, `>> loop N` |
| 기존 도구 수정 | `analyze_prompt` — `autoTag`, `tagMode` 파라미터 추가, `enhancedPromptBlock` 응답 필드 추가 |
| 서버 설정 변경 | `index.js` — `server.prompt()` 도메인별 프리셋 3종 추가 등록 |
| 구현 의존성 | `>> loop`는 `>> fix` 내부 로직 완료 후 활성화 |

도구 총 수: 17개(v0.5.0) → **18개(v0.5.1)**

### 4.8 MCP Prompts 프리셋 확장

**목적:** Claude Desktop `/` 메뉴에 도메인별 특화 워크플로우 프리셋을 추가한다. 현재 3개(`quick-analyze`, `deep-analyze`, `project-report`)에서 6개로 확장하여, 사용자가 분석 목적에 맞는 흐름을 한 번의 선택으로 시작할 수 있게 한다.

**추가 프리셋:**

| 프리셋 이름 | 대상 프롬프트 유형 | 동작 |
|------------|-----------------|------|
| `code-review` | 코드 리뷰 요청 | Role + 언어 + 리뷰 포커스 자동 보완 후 분석 |
| `doc-writing` | 문서·보고서 작성 | 독자·목적·형식 누락 요소 집중 분석 |
| `system-design` | 시스템·아키텍처 설계 | 규모·제약·출력 형식 누락 요소 집중 분석 |

**구현 방식:** `index.js`의 `server.prompt()` 등록 블록에 3개 추가. 각 프리셋은 도메인에 특화된 system instruction을 `analyze_prompt`에 전달하여 분석 포커스를 조정한다. 신규 도구 추가 없이 기존 `analyze_prompt` 재사용.

**트리거 (Claude Desktop):**

```
/code-review  → 코드 리뷰용 프롬프트를 붙여넣으세요
/doc-writing  → 문서 작성용 프롬프트를 붙여넣으세요
/system-design → 시스템 설계용 프롬프트를 붙여넣으세요
```

**프리셋 정의 예시 (code-review):**

```javascript
server.prompt('code-review', {
  description: '코드 리뷰 요청 프롬프트 전문 분석 — Role·언어·리뷰 포커스 누락 여부 집중 점검',
  arguments: [{ name: 'prompt', description: '분석할 코드 리뷰 프롬프트', required: true }],
}, ({ prompt }) => ({
  messages: [{
    role: 'user',
    content: {
      type: 'text',
      text: `다음 코드 리뷰 프롬프트를 분석해줘. 특히 역할 정의(Role), 프로그래밍 언어·버전 명시, 리뷰 포커스(성능/보안/가독성)가 포함되어 있는지 집중 점검해줘:\n\n${prompt}`,
    },
  }],
}));
```

### 4.9 개선 제안 클립보드 포맷 출력

**목적:** `analyze_prompt` 응답에서 `enhancedPrompt`가 JSON 객체 내부에 묻혀 있어 복사·재사용이 불편하다. 개선된 프롬프트를 별도의 마크다운 코드 블록으로 독립 출력하여, 한 번의 복사로 바로 사용할 수 있게 한다.

**현재 vs 개선 출력 비교:**

현재 출력 (개선 전):
```json
{
  "score": 43,
  "enhancedPrompt": "너는 시니어 React 개발자야. TypeScript 기반의 로그인 폼..."
}
```

개선 후 출력:
```
📊 종합 점수: 43/100 (D등급)
...
(분석 결과 요약)
...

✨ 개선된 프롬프트 (복사해서 바로 사용하세요):
─────────────────────────────────────────
너는 시니어 React 개발자야. TypeScript 기반의 로그인 폼 컴포넌트를 만들어줘.
이메일/비밀번호 입력, 유효성 검증, 에러 메시지 표시를 포함하고,
Tailwind CSS로 스타일링해줘. 코드와 함께 사용법을 설명해줘.
─────────────────────────────────────────
entryId: h_xxxxxxxxxx_xxxxxx
```

**구현:** `analyzer.js`의 응답 직렬화 단계에서 `enhancedPrompt`를 구분선(`─`)으로 감싼 `enhancedPromptBlock` 문자열 필드로 추가한다. 기존 `enhancedPrompt` 필드는 하위 호환성을 위해 유지.

**`analyze_prompt` 응답 스키마 변경:**

| 필드 | 변경 | 설명 |
|------|------|------|
| `enhancedPrompt` | 유지 | 기존 문자열 그대로 (하위 호환) |
| `enhancedPromptBlock` | 신규 추가 | 구분선 + 프롬프트 텍스트 + entryId 포함 출력용 블록 |

**제약:** `enhancedPrompt`가 없는 경우(분석 점수 90 이상 등) `enhancedPromptBlock`도 생략한다.

### 4.10 `>> last N` — 최근 N번째 항목 참조 (v0.5.2)

**목적:** 현재 `>> re` / `>> v2`는 직전 1건만 참조할 수 있다. `>> last N`을 추가하여 세션 내 최근 N번째 분석 결과를 entryId 조회 없이 바로 지목할 수 있게 한다.

**트리거 문법:**

| 트리거 | 동작 |
|--------|------|
| `>> last 1` | 직전(가장 최근) entryId 참조 (`>> re`와 동일) |
| `>> last 2` | 2번째 이전 entryId 참조 |
| `>> last N` | N번째 이전 entryId 참조 (최대 10) |

**동작 흐름:**

```
사용자: "이 프롬프트 분석해줘: ... >> last 2"
    │
    ▼
[MCP Server] >> last 2 감지
    │
    ├── session.recentEntryIds 에서 인덱스 1 (0-based) 항목 조회
    │
    ├── 항목 없으면: ⚠️ 세션 내 2번째 이전 분석이 없습니다 (현재 N건)
    │
    └── 있으면: "2번째 이전 entryId `h_xxx` 를 parentId로 설정했습니다."
```

**세션 링 버퍼 설계:**

`session.js`에 `recentEntryIds` 배열(최대 10개)을 추가하여 분석할 때마다 앞에 삽입한다.

```javascript
// session.js 변경
const sessionCtx = {
  lastEntryId: null,      // 기존 (>> re 호환용)
  lastProjectId: null,
  recentEntryIds: [],     // 신규: 최대 10개 링 버퍼
};

export function pushRecentEntryId(id) {
  sessionCtx.recentEntryIds.unshift(id);
  if (sessionCtx.recentEntryIds.length > 10) {
    sessionCtx.recentEntryIds.pop();
  }
  sessionCtx.lastEntryId = id; // 기존 호환
}

export function getRecentEntryIdAt(n) {
  // n은 1-based: 1 = 가장 최근
  return sessionCtx.recentEntryIds[n - 1] ?? null;
}

export function getRecentEntryCount() {
  return sessionCtx.recentEntryIds.length;
}
```

**분석 핸들러 변경 (`handlers/analysis.js`):**

```javascript
// >> last N 트리거 처리
const lastMatch = /^>>\s*last\s+(\d+)$/i.exec(trimmed);
if (lastMatch) {
  const n = parseInt(lastMatch[1], 10);
  const targetId = getRecentEntryIdAt(n);
  const count = getRecentEntryCount();
  if (!targetId) {
    return `⚠️ 세션 내 ${n}번째 이전 분석이 없습니다. (현재 ${count}건 기록됨)`;
  }
  return [
    `${n}번째 이전 entryId \`${targetId}\` 를 parentId로 설정했습니다.`,
    `새 프롬프트를 입력해 주세요 (parentId: \`${targetId}\` 로 자동 연결됩니다).`,
  ].join('\n');
}
```

**`>> re` 하위 호환:** `>> re` / `>> v2`는 `getRecentEntryIdAt(1)`로 통일하여 동일 동작 유지.

**제약 조건:**
- 링 버퍼는 서버 재시작 시 초기화 (세션 기반, 영속성 없음)
- `N > 10`이면 즉시 에러 반환 (링 버퍼 한계 안내)
- `N < 1`이면 즉시 에러 반환
